# Manual Usage Flow

## Purpose

This document explains how to use Pain Timing Map manually.

The goal is to let a user drop a weak signal, small pain, feature announcement, repo trend, or vague discomfort into the system without needing to fully analyze it first.

## Core Rule

The user does not need a fully formed insight.

A light signal is enough.

## Step 1: Light Capture

User provides:

- What I noticed:
- Where I saw it:
- Why it caught my attention:
- Is it urgent? Yes / No / Unknown

Examples:

- "This repo is suddenly popular."
- "This feature announcement feels important."
- "I keep seeing people complain about this."
- "This feels like noise, but I do not want to throw it away."

## Step 2: Signal Recheck

The signal is converted into:

- Source Type
- Surface Claim
- Hidden Pain Hypothesis
- Affected User Species
- Pain Direction
- Bottleneck Migration
- Target Window
- Pre-Signals to Watch
- Payment Reason
- Pull-Forward Risk
- Gate

## Step 3: Output Card

A user should receive:

- Signal
- Hidden Pain
- Who feels it first
- Bottleneck Migration
- Timing
- Pre-Signals
- Gate

## Step 4: Accumulation

Each reviewed signal may be saved in your own notes or workspace.

For a public-safe demo of accumulated signals, see `examples/public_timing_batch_example.md`.

## Step 5: Promotion Path

```text
Light Signal
-> Signal Recheck
-> Signal Log
-> Pain Cluster
-> Evidence Candidate
-> Decision Material Record only with explicit human approval
```

## Current Gate

Allowed:

- manual capture
- manual Signal Recheck
- logging reviewed signals
- clustering discussion

Not allowed:

- automatic search
- scraping
- API intake
- automation
- implementation
- pricing
- release
- public posting
